Global Invoice Parser

A small Rails-based workflow that converts visual or structured invoices into a country-neutral canonical model, proves universal arithmetic, sends uncertain fields to review, and exports approved data. Regional tax semantics and e-invoice conformance are added as independently tested capability packs.

1. Product decision

Build the narrow global core

The differentiator is not generic OCR. It is a fast, auditable workflow that preserves evidence, reconciles amounts, prevents silent errors, and tells operators exactly which capability is supported.

Do not claim “all regions”

The canonical model can represent invoices globally, but regional identifiers, tax semantics, legal requirements, and structured formats must be versioned and benchmarked separately.

Primary launch risk

A broad promise creates unsafe auto-acceptance. Unknown format, language, country, or rule-pack combinations must be reviewed or quarantined rather than guessed.

  • Non-English unattended acceptance
  • Certified Peppol, UBL, CII or government-portal transmission
  • Direct ERP posting, payment execution or ledger coding
  • Blanket support for every regional identifier or format

3. Global capability model

Three independent layers

LayerResponsibilityExamplesFailure behavior
Source syntaxDetect and parse the document container or structured payloadVisual PDF, image, UBL, CII, Factur-X, unknown XMLUse deterministic parser where tested; otherwise quarantine or visual route
Canonical invoiceRepresent parties, identifiers, lines, taxes, totals, references and evidence without country-specific columnsCanonical Invoice Schema v2Schema-invalid output never proceeds
Region rule packApply jurisdiction-specific identifier and tax checks only when supported and selectedglobal_generic_v1, experimental India GST, future EU/US packsUnsupported semantics remain unverified and require review

Money representation

All monetary and numeric business values are decimal strings. Currency precision is resolved from a maintained registry; the core must handle zero-, two-, and three-minor-unit currencies without forcing two decimal places.

Generic tax representation

Taxes are rows with type, component, jurisdiction, category, rate, taxable amount, tax amount, exemption and payable effect. VAT, GST, sales tax, withholding, duties and local components do not require new core columns.

5. Milestones to the first live test

M0 — Pilot contract

Choose two pilot teams, freeze supported inputs/outputs, collect permissioned documents, define critical errors and approve retention/subprocessors.

Exit: signed supervised export-only pilot.

M1 — Canonical core

Implement Schema v2, decimals/currency registry, generic identifiers/taxes, universal validators, duplicate detection and generic exporters.

Exit: USD/EUR/INR/JPY/KWD fixtures pass without country columns.

M2 — Intake and extraction

Secure upload, format detector, structured-payload inspection, provider adapter, provenance, repair limit and benchmark runner.

Exit: frozen holdout is reproducible and unsupported files quarantine safely.

M3 — Review and acceptance

Batch states, risk queue, evidence viewer, canonical editor, locale/profile override, immutable approved revision and approved-only export.

Exit: operator completes a 50-document batch with no unsafe auto-acceptance.

M4 — Live readiness

Authentication, tenant isolation, private storage, purge verifier, content-free logs, quotas, backups, restore drill and security checks.

Exit: privacy, deletion, isolation, recovery and spend controls pass.

M5 — Closed MVP test

Run 500+ documents over two weeks, classify corrections, measure time/cost/precision, regress the frozen holdout, and make a go/iterate/stop decision.

Exit: product and business gates pass.

Post-MVP M6

Select exactly one evidence-backed capability: a demanded regional tax pack, a tested ERP adapter, or read-only structured-invoice support. Do not parallelize country work before the pilot identifies the highest-value path.

6. Release gates

Quality

  • ≥99.5% precision on unattended high-risk fields
  • Zero escaped critical monetary errors in pilot
  • ≥97% core-header exact match on clean digital PDFs
  • ≥92% on the mixed English holdout
  • No critical/high deterministic finding auto-accepts

Operational and business

  • ≥50% operator time saved, or stronger repeatable value in one segment
  • Deletion verifier and restore drill pass
  • No document content in logs, exceptions or job arguments
  • Spend ceiling and retry/circuit-breaker controls active
  • At least one pilot requests continued use
Capability rule: every profile is separately labelled for ingest, extraction, arithmetic validation, regional semantics, export and structured conformance. “Supported” is permitted only after a frozen holdout and explicit release gate pass.

7. Document audit and corrections

Removed from the core

  • GSTIN, CGST, SGST, IGST and IRN columns
  • Two-decimal-only assumptions
  • India-first product positioning
  • Tally as a mandatory export

Added

  • Canonical Invoice Schema v2
  • Source format registry
  • Region profile schema and capability matrix
  • Multicurrency and multi-country fixtures
  • Milestone/issue backlog with exit gates

Still deliberately constrained

  • English is the first benchmark language
  • Regional packs start disabled or experimental
  • Structured formats are detection-first
  • Exports are generic until demanded and tested

9. Recommended starting sequence

  1. Complete M0 before implementing UI.
  2. Implement Schema v2 and universal validators as pure Ruby modules.
  3. Benchmark one visual provider against the frozen holdout.
  4. Build only the review actions required to complete a pilot batch.
  5. Deploy invite-only after M4 evidence passes.
  6. Use M5 corrections to choose the first real regional or structured capability.
